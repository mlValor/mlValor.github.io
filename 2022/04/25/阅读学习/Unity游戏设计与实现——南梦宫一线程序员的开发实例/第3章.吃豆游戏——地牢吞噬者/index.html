第3章.吃豆游戏——地牢吞噬者

发表于2022-04-25|更新于2022-08-11|阅读学习Unity游戏设计与实现——南梦宫一线程序员的开发实例
|字数总计:2.2k|阅读时长:6分钟|阅读量:

1.平滑的网格移动

1.1 按键输入的改善

①.按下按键后，角色不会立即改变方向，而是首先探测是否可以沿该方向前进。如果无法前进，则保持原方向继续运动，直到到达拐角处才改变方向。

②.不过，任何时候都可以向后改变方向（其实就是反向，并不是只指代向后）。

③.只有在通过网格的时候才能改变方向。

1.2 穿过网格的时机

假设砖块尺寸为1。当角色坐标发生整数值变化的时候，就是穿过网格的时机。

比如角色坐标：5.1->5.8->6.5

此时的网格坐标：5->5->6

当网格坐标符合整数值变化的情况，角色改变方向。不仅如此，因为宝石是放在网格上的，所以“穿过网格边界=拾取了宝石”，这样设计的处理速度比使用碰撞检测更有优势。

在穿过网格边界的时候，比较用户输入的方向和目前正在行走的方向，如果二者相反，则将角色朝向旋转180度。

1.3 总结

角色操作性对于任何游戏来说都是非常重要，适当添加操作辅助，角色的操作性说不定就会得到很大的改善。

2.地图数据

2.1 关卡数据制作

因为数据结构比较简单，数据量不大，所以主要使用文本文件的格式。

采用文本文件格式有以下几个优点：

  • 便于理解数据结构（容易查看）
  • 便于修改
序号 标识符 说明 描述
（1） c 宝石 骑士拾取的宝石。将宝石全部收集完毕则过关。
（2） * 砖块 砖块。排列砖块以生成迷宫。砖块除了被用作显示迷宫的模型外，还需要具备碰撞检测的功能。
（3） p 骑士 骑士开始的位置。关卡开始时骑士所在的地方。每个关卡中都必须有一个开始位置。
（4） 1~4 幽灵（共4种，可以放置宝石） 幽灵。从1到4，每个数值分别代表一种性格的幽灵。
（5） s 剑 剑。拾起剑后能进行一次反击。
（6） t 宝箱 宝箱。获取宝箱后将获得得分奖励。游戏开始后，每经过一定的时间将出现一次。
（7） x 空白处 空白区域。用于制作外墙的周围以及被墙壁包围起来的空间。

各个文字之间用逗号隔开，这种用逗号把文字分隔开的文件格式叫作CSV（Comma Separated Values）。

之后将地图数据的文本文件作为文本资源（TextAsset）添加到预设中即可。

2.2 读取地图数据

①.对文本文件整体以行为单位进行分割。利用Split方法将换行符指定为分割符，就能够得到被分割开的各行文本的string数组。

②.文本中表示对象的字符已经被逗号分隔开了。此处继续用逗号分隔开文本并将其放入string数组中。这样全体文本就被分解成了处理的最小单位。这个最小单位我们称之为Token，这里Token = 1个字符。

③.第一行中记录了地图的横向和纵向长度。分别取得横向和纵向的尺寸，创建用于存储地图数据的数组。到这里地图数据的读取就完成了。

2.3 创建地图

①.提前求出地图单元的坐标。除了墙壁的Y坐标与其不同，其他对象的位置坐标都和它一起。

②.根据文本资源生成的地图数据是一个二维数组，可以直接使用XZ的坐标值作为数组的索引来访问数据。

  • “墙”部分用于生成墙的对象。
  • “骑士”部分用于在游戏启动时生成对象。这里提前记录下坐标。
  • “宝箱”部分也做同样的处理。
  • 地图数据为数字时生成幽灵。生成的幽灵性格因数字而异。因为各种性格的幽灵只能生成1只，所以这里需要记住每种幽灵的初始位置。在用作数组的索引时，不要忘记把字符转换为整数。

2.4 总结

可以利用unity编辑器对功能进行扩展，虽然过度关注编辑器的功能扩展有点本末倒置，但是一个好用的工具确实能将制作效率提高数倍。

3.摄像机变焦功能的运用

3.1 相机调整

调整策略：

  • 砍向幽灵的瞬间，变焦推进摄像机，将骑士放大显示在画面上。

  • 然后再拉远摄像机，恢复到一般情况下的状态。

3.2 视野控制和缓动

摄像机的推进和拉远都是在控制“视野”变化，除此之外，还有一种被称为“缓动”（easing）的用来改变动画速度的方法。

视锥体：场景中能够被渲染到画面中的范围被称为视锥体。是一个离摄像机越远开口越宽的四角锥形几何体。视野越大，视锥体的开口就越大，能看见的范围就越大。

一般情况，打算进行大范围渲染时可以使用大视野，如果想放大单个角色进行观察则使用小视野。另外，逐渐缩小视野的过程称为推近（zoom in），相反地，将视野放大的过程则称为拉远（zoom out）。

移动摄像机和变焦推近这两种方法对背景的展现效果有很大的不同。

缓动：缓动效果将改变动画运行的速度，使其不再均匀地变化。

3.3 总结

摄像机的视野对游戏画面的效果有很大影响。即便是相同的角色模型，根据摄像机的设定的不同，显示出来的效果也有很大不同。

4.幽灵的AI

4.1 概要

吃豆游戏中控制好进退的时机非常重要。无论是玩家还是敌方，在游戏的过程中都要开动脑筋，揣测对方的习惯，分析他的行为，以避免在狭窄的迷宫内被敌人追上。

当然一味的增加难度只会让游戏变得无趣。如果敌方也会偶尔出现失误，就会让玩家感觉对方是一个生物而非冰冷机器，从而使游戏更具真实感。

4.2 跟踪算法

首先说明4种类型的思考过程。

①.追赶型。幽灵一直在骑士后面追赶。思考过程非常简单，制作起来也没什么难度。但另一方面，这也不是一种聪明的做法。

②.埋伏等待型。幽灵提前在骑士将要到达的地方埋伏等待。针对那些单纯地认为“敌人会从后面追来”的玩家，这是一种有效的策略。

③.包围攻击型。配合追赶型幽灵夹击骑士。考虑到迷宫内的通道非常狭窄，对玩家来说，这种类型的幽灵可能是非常麻烦的对手。

④.随机型。和骑士的位置无关，幽灵随机改变方向。因为其行动毫无规律无法预测，所以如果幽灵全部都是这种类型，游戏就会丧失分析决策的乐趣。不过适当地加入一些随机因素，则会让玩家产生紧张感。

接下来讲解决定移动方向的流程。

追赶型幽灵：

①.首先算出从自己的位置指向骑士位置的向量。

②.因为迷宫只能上下左右移动，所以必须从中选择最接近目标的方向移动，通过比较刚才算出的向量的X分量和Z分量计算得出。

埋伏等待型幽灵：

与“追赶型”并无太大差别，仅仅是移动的目标位置不同。“追赶型”的目标是骑士所在的位置，而“埋伏等待型”的目标是骑士前方不远处的位置。

包围攻击型幽灵：

包围攻击型向着以骑士为中心和追赶型对称的位置移动。移动的方法相同。

随机型幽灵：

和骑士位置没关系了，全靠随机数来决定前进的方向。

4.3 总结

《地牢吞噬者》中的思考过程，相对于那些使用真正的AI的游戏来说略显简单。可以尝试实现一些与众不同的思考过程，比如幽灵从骑士处逃离，不知为何去追赶幽灵同伴等。
